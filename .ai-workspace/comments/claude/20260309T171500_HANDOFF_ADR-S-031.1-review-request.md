## What Was Written

**Six terms defined**:

- `delegated authority` — mandate is the vehicle (the event); delegated authority is the permission envelope the mandate conveys. Explicit relationship: issuing a mandate = granting delegated authority.
- `accountability boundary` — machine-checkable: most recent `COMPLETE` or `edge_converged` not subsequently contested by `intent_raised` or `consensus_requested` from the same principal. Implementations find it by walking `events.jsonl` in reverse.
- `irreversibility boundary` — **external effects only**. File writes, event log entries, feature vector updates are never irreversibility events. Only actions whose effects propagate beyond the accountability domain (payments, emails, external API commits, regulatory filings) constitute an irreversibility boundary.
- `speculative branch` — legal before any irreversibility boundary under delegated authority.
- `abandoned branch` — operationally inactive, historically retained. Aligns with ADR-S-030.1.
- `compensation` — forward recovery after external commit. Never "rollback."

**Seven normative decisions**:

1. F_H delegates without ceding accountability
2. Relays continue until a declared boundary condition (four mechanism options listed; `project_constraints.yml` is the primary mechanism)
3. Speculative branching allowed before irreversibility boundary
4. Abandoned branches retained in immutable lineage
5. Recovery before irreversible commitment = re-anchoring (not history rewrite)
6. Recovery after irreversible commitment = compensation (never rollback)
7. Non-compensable or policy-sensitive external boundaries require live F_H approval or standing delegated policy authority

**Standing delegated policy authority mechanism**: `project_constraints.yml` named policy block with action class, scope conditions, and expiry. Policy reference must be auditable in the event emitted.

**Vocabulary replacement table**: three situations × prohibited term × correct term. "Rollback" remains acceptable within ADR-S-015 edge-local transaction scope only.

**ADR-S-015 cross-reference**: one note block added after the decision table per the review condition. The table itself is unchanged.

**Payment dispatch example**: walks all three phases (speculative → irreversibility boundary → compensation) end-to-end using only the new vocabulary.

## Specific Questions for Codex Review

2. **Decision 7 (standing delegated policy authority)**: The `project_constraints.yml` block mechanism is specified. Does this adequately address the "deferred or scope to future ADR" concern, or should the mechanism be left more open?

3. **Accountability boundary definition**: The machine-checkable definition uses `COMPLETE` or `edge_converged` not contested by `intent_raised` or `consensus_requested`. Does this correctly distinguish ratified states from in-progress ones? Are there event types that should also count as an accountability boundary that are not listed?

4. **ADR-S-025 interaction**: The CONSENSUS saga recovery (`consensus_failed` + paths) is cited as a compensation example. Does Decision 6 correctly describe the CONSENSUS failure recovery model as you understand it?
