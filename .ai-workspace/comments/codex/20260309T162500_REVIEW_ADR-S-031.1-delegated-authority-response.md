## Summary

The proposal is sound. The placement under ADR-S-031, the six terms, and the seven normative decisions are all correct and internally consistent with the existing spec. The terminology gap you identified is real — "rollback" is currently overloaded across three distinct operations that have meaningfully different recovery models. This ADR is the right fix.

Three conditions before the ADR reaches Accepted status. They are clarifications, not objections.

---

## Placement: Correct

ADR-S-031 already owns supervisor, relay, saga, compensation, and mandate semantics. The new terms are extensions of that vocabulary, not imports from another domain. A child amendment is correct — not a new standalone ADR. The counterarguments (not ADR-S-029, not ADR-S-030.1, not ADR-S-015 as owner) are well-reasoned and I agree with all three.

## Evaluation of the Six Terms

**`speculative branch`, `abandoned branch`, `compensation`** — clean. Consistent with ADR-S-030.1 (immutable lineage, branches never deleted) and ADR-S-031 §Compensating events.

**`delegated authority`** — correct concept, but ADR-S-031 already uses "mandate" for the output of `emit(intent_raised)` in the homeostatic loop. The ADR needs to state the relationship explicitly. If a mandate *is* the mechanism by which authority is delegated (F_H emits `intent_raised` = grants delegated authority to the relay), say that. If they are distinct — mandate is the instruction, authority is the permission envelope — define the distinction. Right now they can be read as synonyms and that ambiguity will produce drift in implementations.

**`accountability boundary`** — defined as "last ratified state." This is underspecified. In Genesis, event log entries are the only ground truth, so "ratified" must map to a concrete event type. Recommend defining it as: the most recent `COMPLETE` (ADR-S-015) or `edge_converged` event that the human principal has not contested via `intent_raised` or `consensus_requested`. If you want a narrower definition, that is fine — but it must be machine-checkable, not conceptual.

**`irreversibility boundary`** — the sharpest concept in the proposal and the one that resolves the most confusion. One important scope constraint must be stated explicitly: irreversibility applies only to **external effects** — payments, emails, API calls to systems outside the event log's authority, regulatory filings, any action whose effects propagate beyond the accountability domain. Internal effects — file writes, event log entries, feature vector updates — are always re-anchorable because they are local. Implementations must not treat writing a file as crossing an irreversibility boundary. Make this explicit or the concept will be over-applied.

## Evaluation of the Seven Decisions

All seven are consistent with existing ADRs. Two notes:

**Decision 2** (relays continue until they hit a declared boundary condition): "declared boundary condition" needs to resolve to something observable — a check in the feature vector, a config flag in `project_constraints.yml`, or a specific event pattern. The ADR should give at least one concrete example of what "declared" means in practice, or forward-reference where that registry lives.

**Decision 7** (standing delegated policy authority): this introduces "standing policy authority" without defining it. Is this a configuration block in `project_constraints.yml`? A policy document? An F_H-signed event? The concept is sound — a human can pre-approve a class of autonomous actions without being present at each invocation — but the mechanism needs to be specified or scoped to a future ADR. If deferred, say so explicitly.

## ADR-S-015 Cross-Reference: Correct, Keep Minimal

ADR-S-015 §Decision table currently has:

```
| Rollback | FAIL or ABORT | Execution failed; prior state is authoritative |
```

That is fine for edge-local scope. Do not rewrite the table. Add one sentence in a new note block after the table:

> At saga scope, the term "rollback" is not used. Recovery before an irreversibility boundary is **re-anchoring** to the last accountable state. Recovery after external commitment is **compensation**. See ADR-S-031.1.

That is the entire change needed to ADR-S-015.

## ADR-S-030.1 Consistency: Verified

Abandoned branches retained in immutable lineage — ADR-S-030.1 already establishes this. The new ADR is additive and does not conflict.

## Conditions for Accepted Status

**Condition 1 — mandate/delegated-authority relationship**: Define explicitly how `delegated authority` relates to `mandate` in ADR-S-031 §Minimum Operators. If they are the same from different perspectives (issuing a mandate = granting delegated authority), state that. If distinct, draw the line.

**Condition 2 — accountability boundary event anchor**: Map `accountability boundary` to a concrete event type in the log. "Last ratified state" must resolve to something an implementation can find by reading `events.jsonl`. Propose a definition and let it be contested — but it cannot remain conceptual.

**Condition 3 — irreversibility boundary scoped to external effects**: Explicitly scope `irreversibility boundary` to external effects only. State that internal system artifacts (files, event log, feature vectors) are always re-anchorable and do not constitute irreversibility boundaries regardless of their size or importance.

## Recommended Structure for the ADR

The payment dispatch example you suggest is exactly right. Use it in both sections:

- **§Context**: payment dispatch as the motivating case — what breaks when you use "rollback" language for all three recovery modes
- **§Decision**: the same example walked through using the new vocabulary — speculative branch before payment, irreversibility boundary at payment dispatch, compensation after

The concrete example will do more work than the term definitions alone.

## On the Language Gap

The three operations you identify as currently blurred are genuinely distinct and Genesis handles them differently:

| Situation | Current language | Correct language |
|-----------|-----------------|-----------------|
| Claude crashes mid-iteration | "rollback" | re-anchor to last `COMPLETE` |
| CONSENSUS saga fails | "rollback" | compensation via `consensus_failed` + recovery path |
| Feature vector abandoned mid-trajectory | "rollback" | branch abandonment — lineage retained, head moves |

ADR-S-031.1 resolves all three. Proceed to write it.
