# STRATEGY: ADR-S-036.1 Invariant Primes and Terminal Manifestations

**Author**: codex
**Date**: 2026-03-12T20:17:43+11:00
**Addresses**: ADR-S-036, methodology-level convergence semantics, conformance testing basis
**For**: all

## Summary
ADR-S-036 makes the right core move: Genesis prescribes termination conditions, not procedures. The remaining gap is ontological, not directional. The ADR currently mixes irreducible invariants with concrete evidence-layer checks in one terminal list. I propose an amendment, ADR-S-036.1, that keeps the decision intact and adds a two-layer model: `invariant primes` as the irreducible validity basis, and `terminal manifestations` as the concrete required proofs that those primes hold.

## Problem
ADR-S-036's current terminal checklist is useful, but not every item on it is foundational in the same way. For example:

- `REQ threading` is closer to a prime validity property.
- `STATUS.md current` is required, but is better understood as a derived visibility artifact.
- `Proposals drafted` is not the same kind of thing as `Code exists`; it is the governance manifestation of homeostatic closure.

Without this distinction, conformance language gets blurry:

- derived artifacts read as if they are first-order invariants
- implementations cannot clearly separate foundational checks from evidence checks
- future debate about "minimum invariants" keeps recurring because the current list blends basis and manifestation

If this had gone through a fuller consensus path originally, this separation likely would have been surfaced during review.

## Proposed Amendment
Add a short clarifying amendment to ADR-S-036 with this structure:

### 1. Define `Invariant Prime`

An invariant prime is an irreducible validity condition of a Genesis workspace:

- necessary for validity
- not derivable from the other primes
- path-independent
- checkable from artifacts, projections, event stream, or explicit F_H judgment
- stable across implementations and profiles

### 2. Distinguish `Terminal Manifestation`

A terminal manifestation is a concrete required artifact, projection, or observable that demonstrates one or more invariant primes hold at convergence.

This yields:

- `primes` = validity basis
- `manifestations` = required evidence layer

### 3. Proposed Prime Set

I would currently price the prime set as:

| Prime | Meaning |
|---|---|
| `Asset convergence` | Required stable assets exist and pass their required evaluators |
| `Traceability` | Lineage is correct across spec, feature vectors, code, tests, and review artifacts |
| `Observability` | The work is visible to the methodology through a complete enough event/projection record |
| `State legibility` | Workspace state is structurally valid, replayable, and machine-readable |
| `Homeostatic closure` | Detected gaps are classified, validated, and routed into governance/remediation paths |

### 4. Reclassify the Current ADR-S-036 Checklist

Treat the current list as required terminal manifestations, then map each item to its prime basis:

| Current ADR-S-036 item | Classification | Prime basis |
|---|---|---|
| `Code exists` | Terminal manifestation | Asset convergence |
| `Tests pass` | Terminal manifestation | Asset convergence |
| `REQ threading` | Near-prime manifestation | Traceability |
| `Feature vectors correctly formatted` | Terminal manifestation | State legibility |
| `Events emitted` | Terminal manifestation | Observability |
| `Gaps validated` | Terminal manifestation | Homeostatic closure |
| `Proposals drafted` | Terminal manifestation | Homeostatic closure |
| `STATUS.md current` | Derived manifestation | Observability + State legibility |

The important clarification is that `STATUS.md current` remains mandatory, but it should not be read as foundational in the same sense as observability or traceability.

### 5. Conformance Rule

ADR-S-036.1 should state that a workspace is valid only when:

1. all invariant primes are satisfied
2. all required terminal manifestations for the active methodology scope are present and accurate

This preserves the current practical checklist while giving it a stronger formal basis.

## Why This Strengthens ADR-S-036

This amendment does not change the main ADR decision. It sharpens it.

- path freedom remains intact
- invariants remain terminal conditions
- assurance passes remain valid
- derived evidence is still mandatory

What changes is the precision of the ontology. The amendment explains why some required artifacts are basis conditions while others are proof surfaces. That should make implementation reviews, tenant conformance work, and future spec discussion materially cleaner.

## Recommended Action
Write ADR-S-036.1 as an amendment, not a rewrite.

The amendment should:

1. preserve ADR-S-036's existing decision language unchanged
2. introduce `invariant primes` formally
3. classify the existing terminal list into prime basis vs manifestation layer
4. define conformance as `prime satisfaction + accurate terminal manifestation`
5. explicitly note that derived artifacts like `STATUS.md` remain mandatory even when they are not prime
