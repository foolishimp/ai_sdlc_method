# ADR-S-035: Genesis Topology Language (GTL) — Python DSL over Static File Structure

**Status**: PROPOSED
**Date**: 2026-03-11
**Authors**: foolishimp (steering); Claude, Codex (cross-tenant design convergence)
**Scope**: Specification-level — applies to all Genesis implementations
**Marketplace convergence**: Claude `20260311T105052`, `20260311T110500`, `20260311T111500`; Codex `20260311T215329`, `20260311T215629`, `20260311T220312`

---

## Context

Genesis currently packages its methodology as a bundle of static files that conflate three distinct concerns:

1. **Methodology definition** — what graph topology, which operators, which compositions
2. **Project configuration** — project-specific values and constraints
3. **Runtime state** — the event log, feature vectors, workspace artifacts

The static files (`graph_topology.yml`, `edge_params/*.yml`, `profiles/*.yml`) are simultaneously the authoring surface and the execution contract. This produces three compounding problems:

**Topology is not parameterisable.** `graph_topology.yml` is a global definition shared by all projects. Profiles are skip-lists (`graph.skip: [edge_a, edge_b]`) — they can omit edges but cannot add domain-specific ones, express inheritance, or verify that the resulting graph is valid. A fintech compliance gate, a mobile app review step, and a data pipeline quality check all require hand-editing the shared YAML directly.

**Methodology and project are conflated.** Changing the methodology means editing the same files as project state. Versioning is per-bundle, not per-layer.

**Higher-order composition is inexpressible at the methodology level.** Named compositions (ADR-S-026) are `{macro, version, bindings}` structs in YAML — they are patterns the LLM recognises, not executable constructs the compiler validates. Sequences, parallel edges, retry policies, and conditional branches require engine changes, not methodology changes.

The Helm analogy is precise: Helm solved the same problem for Kubernetes by separating *charts* (parameterisable methodology programs) from *values* (project configuration) from *releases* (deployed runtime state). Genesis currently operates at the pre-Helm level.

---

## Decision

**Genesis packages become GTL programs, not file bundles.**

The **Genesis Topology Language (GTL)** is a Python internal DSL that replaces the static configuration files as the canonical authoring surface. The existing YAML files (`graph_topology.yml`, `edge_params/*.yml`, `profiles/*.yml`) become compiled materialisations — they are generated by the GTL compiler and remain the execution contract unchanged.

**Phase 1 (this ADR):** GTL as canonical authoring surface; compiler emits today's YAML runtime contract; all existing consumers (genesis_monitor, engine, installer, verifier) unchanged.

**Phase 2 (future ADR):** After Phase 1 is proven and the spec reprices the filesystem contracts, the engine optionally consumes the GTL program directly (live topology traversal). Phase 2 is explicitly out of scope here.

The three-layer separation:

| Layer | Authored as | Role |
|-------|------------|------|
| `genesis.gtl.py` (methodology) | GTL Python DSL | Graph topology + operators + compositions |
| `project_constraints.yml` (project) | YAML | Project-specific values — unchanged |
| `.ai-workspace/` (runtime) | Event log + vectors | Append-only source of truth — unchanged |

---

## The GTL Semantic Model

GTL is grounded in the four primitives of the Asset Graph Model:

```
Graph    → declared with Graph class; topology of typed assets and admissible transitions
Iterate  → expressed as Composition blocks; iterate(Asset, Context[], Evaluators) stays unchanged
Evaluators → declared as Operator objects; convergence rules on each Edge
Spec+Context → expressed as Workspace and Tenant declarations; resource locators replace hardcoded paths
```

The compiler is a pure F_D function (deterministic, no LLM required): it reads a `genesis.gtl.py` file, validates the semantic model, and emits the current YAML runtime artifacts. This matches the composition compiler pattern defined in ADR-S-026 §3.

### 1. Operator Declarations

An `Operator` names a functional unit, assigns it a category (F_D / F_P / F_H), and binds it to an implementation.

#### Syntax

```python
name = Operator(
    name: str,
    category: F_D | F_P | F_H,
    bind: str,
    params: dict = None,
    description: str = None,
)
```

#### Bind forms

| Form | Category | Meaning |
|------|----------|---------|
| `"python -m pytest {test_path} -v"` | F_D | Shell command; `{param}` resolved from edge context |
| `"genesis::{function}"` | F_D | Built-in genesis evaluator function |
| `"mcp::{tool_name}"` | F_P | MCP tool dispatch via the fold-back protocol (ADR-023) |
| `"agent::{prompt_key}"` | F_P | LLM agent invocation; prompt loaded from edge guidance |
| `"interactive"` | F_H | Human gate; blocks until human approves or dismisses |
| `"consensus::{quorum}"` | F_H | Multi-stakeholder CONSENSUS functor (ADR-S-025) |

#### Validation rules

- F_H operators **must** use bind form `interactive`, `consensus::*`, or a custom F_H implementation
- F_D operators **must not** use bind form `agent::*` or `mcp::*`
- `name` must be unique within the operator registry of a given Graph
- Operators are scoped to the Graph they are declared in; they may be imported from a base Graph via `extends`

#### Example

```python
# Deterministic operators
pytest        = Operator("pytest",         F_D, bind="python -m pytest {test_path} -v")
coverage      = Operator("coverage",       F_D, bind="python -m pytest --cov={src_path} --cov-fail-under={threshold}",
                          params={"threshold": 80})
req_keys_tagged = Operator("req_keys_tagged", F_D, bind="genesis::check_req_tags")
all_tests_pass  = Operator("all_tests_pass",  F_D, bind="genesis::all_tests_pass")

# Agent operators
llm_gap_analysis = Operator("llm_gap_analysis", F_P, bind="mcp::claude-code-runner")
req_extraction   = Operator("req_extraction",   F_P, bind="agent::requirements_extraction")

# Human operators
human_approval   = Operator("human_approval",   F_H, bind="interactive")
architecture_review = Operator("architecture_review", F_H, bind="consensus::quorum=2/3")
```

### 2. Edge Declarations

An `Edge` parameterises a single graph transition: its evaluator checklist, convergence rule, encoding, and guidance.

#### Syntax

```python
Edge(
    source: str,
    target: str,
    evaluators: list[Operator | RequiredEvaluator | OptionalEvaluator],
    convergence: ConvergenceRule = all_required,
    co_evolve: bool = False,
    human_required: bool = False,
    encoding: F_D | F_P | F_H | None = None,
    source_analysis: list[Operator] = None,
    context: EdgeContext = None,
    guidance: str = None,
)
```

#### Evaluator wrappers

```python
required(op)   # evaluator must pass for convergence (default for bare Operator)
optional(op)   # evaluator runs but failure does not block convergence
```

#### Convergence rules

| Rule | Meaning |
|------|---------|
| `all_required` | All `required()` evaluators must pass (default) |
| `all_pass` | Every evaluator (including optional) must pass |
| `human_approves` | Convergence gated on a human evaluator passing |
| `any_required` | At least one `required()` evaluator must pass |

#### Co-evolution

`co_evolve=True` declares that `source` and `target` iterate together in the same loop. Both assets are inputs and outputs of every `iterate()` call on this edge. Represented as `source↔target` in the graph topology. The sequence must place a co-evolution edge as a tuple `(source, target)`.

#### EdgeContext

```python
EdgeContext(
    required: list[str],    # context items the agent must load
    recommended: list[str], # context items the agent should load if available
    notes: str = None,
)
```

#### Example

```python
intent_to_requirements = Edge(
    source="intent",
    target="requirements",
    evaluators=[
        required(req_extraction),
        required(Operator("intent_covered",  F_P, bind="agent::check_intent_coverage")),
        required(Operator("keys_testable",   F_P, bind="agent::check_testability")),
        required(human_approval),
    ],
    convergence=human_approves,
    context=EdgeContext(
        required=["intent document", "project_constraints.yml"],
        recommended=["existing ADRs", "domain reference material"],
    ),
    guidance="Extract structured REQ-* keys from intent. Each key must be testable and trace to INT-*.",
)

code_unit_tests = Edge(
    source="code",
    target="unit_tests",
    co_evolve=True,
    evaluators=[
        required(pytest),
        required(coverage),
        required(all_tests_pass),
        required(req_keys_tagged),
    ],
    convergence=all_required,
    encoding=F_D,
)
```

### 3. Graph Declarations

A `Graph` declares the full topology: asset node registry, edges, and sequence. A Graph may extend another Graph, inheriting its nodes, edges, and operators.

#### Syntax

```python
class MyGraph(Graph):
    name: str                            # required; unique identifier
    extends: type[Graph] | None = None   # base graph for inheritance
    description: str = None

    nodes: list[str]             # asset type names; order is informational only
    edges: list[Edge]            # all admissible transitions
    extra_edges: list[Edge] = [] # additional edges added by this subgraph (when extends is set)
    sequence: list[str | tuple]  # canonical traversal order; tuple = co-evolution pair

    operators: list[Operator] = []  # operator registry for this graph
```

When `extends` is set:
- `nodes` is optional — defaults to parent's nodes
- `edges` defaults to parent's edges; `extra_edges` adds to them
- `sequence` must be provided explicitly when the topology changes; otherwise it inherits
- All parent operators are available; new operators declared in the subclass are added

#### Validation rules (compile-time)

1. Every `source` and `target` string in every Edge must be in `nodes`
2. `sequence` must be a valid topological ordering of the edge DAG (no edge `A→B` where B precedes A in sequence)
3. No cycles in the edge graph (DAG invariant)
4. Every co-evolution tuple `(A, B)` in `sequence` must have a corresponding `Edge(source=A, target=B, co_evolve=True)`
5. Every `Operator` referenced in an edge's `evaluators` list must be declared in this graph's `operators` list or inherited from an ancestor
6. If `extends` is set, `extra_edges` must not redefine (same source+target) any edge from the parent without explicit `override=True`

#### Example — standard SDLC graph

```python
class StandardGraph(Graph):
    name = "standard"
    description = "Normal feature development — full decomposition chain"

    operators = [pytest, coverage, all_tests_pass, req_keys_tagged,
                 llm_gap_analysis, req_extraction, human_approval, architecture_review]

    nodes = [
        "intent", "requirements", "feature_decomposition",
        "design_recommendations", "design",
        "module_decomposition", "basis_projections",
        "code", "unit_tests", "uat_tests", "cicd",
    ]

    edges = [
        Edge("intent",                "requirements",          evaluators=[...], convergence=human_approves),
        Edge("requirements",          "feature_decomposition", evaluators=[...], convergence=human_approves),
        Edge("feature_decomposition", "design_recommendations",evaluators=[...]),
        Edge("design_recommendations","design",                evaluators=[...], convergence=human_approves),
        Edge("design",                "module_decomposition",  evaluators=[...]),
        Edge("module_decomposition",  "basis_projections",     evaluators=[...]),
        Edge("basis_projections",     "code",                  evaluators=[...]),
        Edge("code",                  "unit_tests",            evaluators=[pytest, coverage, all_tests_pass],
             co_evolve=True, encoding=F_D),
        Edge("design",                "uat_tests",             evaluators=[...], optional_in_sequence=True),
        Edge("code",                  "cicd",                  evaluators=[...], optional_in_sequence=True),
    ]

    sequence = [
        "intent", "requirements", "feature_decomposition",
        "design_recommendations", "design", "module_decomposition",
        "basis_projections", ("code", "unit_tests"),
    ]
```

#### Example — domain extension

```python
class FintechSDLC(Graph):
    name = "fintech_sdlc"
    extends = StandardGraph
    description = "Fintech — adds SOX compliance gate before code"

    operators = [
        *StandardGraph.operators,
        Operator("sox_controls_present",       F_D, bind="genesis::check_sox_controls"),
        Operator("data_classification_complete", F_D, bind="genesis::check_data_classification"),
    ]

    extra_edges = [
        Edge("design", "compliance_review",
             evaluators=[
                 required(Operator("sox_controls_present",        F_D, bind="genesis::check_sox_controls")),
                 required(Operator("data_classification_complete", F_D, bind="genesis::check_data_classification")),
                 required(Operator("compliance_human_gate",        F_H, bind="interactive")),
             ],
             convergence=human_approves,
             override=False),
    ]

    sequence = [
        "intent", "requirements", "feature_decomposition",
        "design_recommendations", "design", "compliance_review",
        "module_decomposition", "basis_projections", ("code", "unit_tests"),
    ]
```

### 4. Profile Algebra

A **Profile** is a named, validated subgraph expression derived from a Graph via algebraic operators. Profiles are compile-time checked — the compiler rejects any profile that produces an invalid subgraph before emitting YAML.

#### Operators

| Operator | Python form | Meaning |
|----------|------------|---------|
| Subtraction | `G - S` | Remove nodes in set S and all their incident edges |
| Addition | `G + E` | Add Edge objects E; new nodes inferred from edges |
| Intersection | `G & S` | Keep only nodes in S and edges where both endpoints are in S |

`S` is always a Python `set` of node name strings. `E` is a list of `Edge` objects.

#### Subtraction semantics (`G - S`)

```
Input:  Graph G, set of node names S
Output: Profile P
Rules:
  P.nodes    = G.nodes - S
  P.edges    = {e ∈ G.edges | e.source ∉ S and e.target ∉ S}
  P.sequence = [n for n in G.sequence if n ∉ S
                and not (isinstance(n, tuple) and any(x ∈ S for x in n))]
Validation:
  ERROR if P.nodes is empty
  ERROR if P.sequence creates an unreachable start node
  WARNING if removing a node disconnects a required predecessor chain
    (e.g., removing "requirements" from a graph where "feature_decomposition"
     requires it — the compiler flags this as a broken dependency)
```

#### Addition semantics (`G + E`)

```
Input:  Graph G, list of Edge objects E
Output: Profile P
Rules:
  new_nodes  = {e.source for e in E} | {e.target for e in E} - G.nodes
  P.nodes    = G.nodes ∪ new_nodes
  P.edges    = G.edges ∪ E
  P.sequence = G.sequence  (caller must provide sequence_inject to position new nodes)
Validation:
  ERROR if adding E creates a cycle in the edge DAG
  ERROR if new_nodes are introduced but sequence_inject is not provided
  ERROR if any edge in E redefines an existing G edge without override=True
```

#### Intersection semantics (`G & S`)

```
Input:  Graph G, set of node names S
Output: Profile P
Rules:
  P.nodes    = G.nodes ∩ S
  P.edges    = {e ∈ G.edges | e.source ∈ S and e.target ∈ S}
  P.sequence = [n for n in G.sequence if n ∈ S
                or (isinstance(n, tuple) and all(x ∈ S for x in n))]
Validation:
  ERROR if P.nodes is empty
  ERROR if P.edges is empty (intersection produced an isolated node set)
```

#### Profile declaration

```python
# Named profiles via algebra
hotfix  = Profile("hotfix",   StandardGraph - {"feature_decomposition", "module_decomposition",
                                                "basis_projections", "uat_tests", "cicd"},
                  description="Emergency fix — direct path to code↔unit_tests")

poc     = Profile("poc",      StandardGraph - {"module_decomposition", "basis_projections",
                                               "uat_tests", "cicd"},
                  description="Proof of concept — skip decomposition, no CI/CD")

spike   = Profile("spike",    StandardGraph & {"intent", "requirements", "code", "unit_tests"},
                  description="Research — minimal edges, time-boxed")

minimal = Profile("minimal",  StandardGraph & {"code", "unit_tests"},
                  description="Single edge — trivial change or hotfix with known scope")

full    = Profile("full",     StandardGraph,
                  description="All edges including optional — regulated or high-stakes")
```

#### Validation output at compile time

```
Profile 'hotfix' validation:
  Base: StandardGraph (11 nodes, 9 required edges)
  Removed: {feature_decomposition, module_decomposition, basis_projections, uat_tests, cicd}
  Result: 6 nodes, 4 edges — valid subgraph
  Sequence: intent → requirements → design_recommendations → design → (code ↔ unit_tests)
  [OK] No broken predecessor chains
  [OK] Reachable from intent to terminal node
```

### 5. Composition Declarations

A `Composition` is a named, reusable workflow pattern defined in terms of the iterate function and control flow primitives. It corresponds to Level 3 in the ADR-S-026 five-level stack: named compositions that compile to graph edges and edge parameterisations.

#### Syntax

```python
@composition
def name(params...) -> CompositionResult:
    ...
```

The `@composition` decorator registers the function in the composition library, validates its signature, and makes it callable from Edge declarations.

#### Control flow primitives

| Primitive | Signature | Meaning |
|-----------|-----------|---------|
| `iterate(asset)` | `iterate(asset: AssetRef) → IterateExpr` | Single iterate() invocation on asset |
| `co_evolve(a, b)` | `co_evolve(a: AssetRef, b: AssetRef) → IterateExpr` | Co-evolution iterate() |
| `.until(condition)` | `IterateExpr.until(c: Condition) → BoundedIterate` | Loop until condition holds |
| `.timeout(duration)` | `BoundedIterate.timeout(d: str) → TimedIterate` | Add time-box |
| `.on_stuck(action)` | `*.on_stuck(a: Action) → *` | Action when δ unchanged for 3 iterations |
| `.on_timeout(action)` | `*.on_timeout(a: Action) → *` | Action when time-box expires |
| `escalate(op)` | `escalate(op: Operator) → Action` | Escalate to higher evaluator category |
| `fold_back(parent)` | `fold_back(parent: FeatureRef) → Action` | Return result to parent vector |
| `emit(event_type, data)` | `emit(str, dict) → Action` | Emit event to event log |

#### Condition combinators

```python
condition_a & condition_b   # both must hold
condition_a | condition_b   # either must hold
~condition_a                # condition must not hold
```

Built-in conditions: `all_tests_pass`, `coverage(threshold=N)`, `human_approved`, `delta_zero`, `time_elapsed(duration)`.

#### Example compositions

```python
@composition
def tdd_cycle(feature, code_asset, test_asset,
              coverage_threshold: int = 80,
              timeout_duration: str = "2h"):
    """TDD co-evolution: iterate code↔tests until all tests pass and coverage met."""
    return (
        iterate(co_evolve(code_asset, test_asset))
        .until(all_tests_pass & coverage(threshold=coverage_threshold))
        .timeout(timeout_duration)
        .on_stuck(escalate(human_approval))
        .on_timeout(fold_back(feature))
    )


@composition
def review_gate(feature, asset, quorum: str = "1"):
    """Human review gate: block until human approves or dismisses."""
    return (
        iterate(asset)
        .until(human_approved)
        .on_stuck(emit("intent_raised", {"signal_source": "human_gate_stalled",
                                         "feature": feature}))
    )


@composition
def schema_discovery_cycle(feature, data_source, output_schema):
    """Schema discovery: run notebook, extract schema, validate."""
    return (
        iterate(data_source)
        .until(delta_zero)
        .timeout("4h")
        .on_stuck(escalate(human_approval))
    )
```

#### Using compositions in Edge declarations

```python
Edge("code", "unit_tests",
     co_evolve=True,
     evaluators=[pytest, coverage, all_tests_pass],
     execute=tdd_cycle,           # named composition governs iteration control
     convergence=all_required)
```

When `execute` is provided, the composition's control flow overrides the default iterate loop for that edge. The composition's `.until()` condition must be satisfiable by the edge's `evaluators`.

### 6. Tenancy Declarations

Tenants are explicit architectural declarations. A tenant owns a set of implementation paths and is associated with a design variant. The tenancy boundary must not be hidden behind generic path resolution (Codex ADR-S-035 review).

#### Syntax

```python
Tenant(
    name: str,
    impl: PathSpec,
    tests: PathSpec,
    design: PathSpec = None,
    description: str = None,
)
```

`PathSpec` is created with `path("...")` — an absolute or project-relative path string.

#### Example

```python
claude_tenant  = Tenant("claude",  impl=path("imp_claude/"),  tests=path("imp_claude/tests/"),
                         design=path("imp_claude/design/"))
gemini_tenant  = Tenant("gemini",  impl=path("imp_gemini/"),  tests=path("imp_gemini/tests/"),
                         design=path("imp_gemini/design/"))
codex_tenant   = Tenant("codex",   impl=path("imp_codex/"),   tests=path("imp_codex/tests/"),
                         design=path("imp_codex/design/"))
default_tenant = Tenant("default", impl=path("src/"),         tests=path("tests/"))
```

#### Validation rules

- `name` must be unique within a Workspace
- `impl` path must exist at compile time (warning, not error — new projects may not have the directory yet)
- A tenant name must not collide with a node name in the graph (prevents ambiguous resolution)
- At most one `default` tenant per Workspace

### 7. Workspace Declarations

A `Workspace` declares all resource paths for a project. It replaces hardcoded paths in the engine and makes resource resolution inspectable.

#### Syntax

```python
class MyWorkspace(Workspace):
    events   : PathSpec       # append-only event log directory
    features : PathSpec       # feature vector storage
    spec     : PathSpec       # specification/ root
    graph    : PathSpec       # compiled graph_topology.yml location
    agents   : PathSpec       # per-agent working state
    context  : PathSpec       # project_constraints.yml parent dir
    reviews  : PathSpec       # pending proposals queue

    tenants  : list[Tenant]   # explicit tenant declarations (no path aliases)
```

All paths are relative to the project root (the directory containing `genesis.gtl.py`).

#### Example

```python
class ProjectWorkspace(Workspace):
    events   = path(".ai-workspace/events/")
    features = path(".ai-workspace/features/")
    spec     = path("specification/")
    graph    = path(".ai-workspace/graph/")
    agents   = path(".ai-workspace/agents/")
    context  = path(".ai-workspace/context/")
    reviews  = path(".ai-workspace/reviews/pending/")

    tenants = [claude_tenant, gemini_tenant, codex_tenant, default_tenant]
```

#### Default workspace

If no `Workspace` subclass is declared in `genesis.gtl.py`, the compiler uses `DefaultWorkspace` which encodes the current hardcoded paths. This is the backward-compatible default — existing projects without a `genesis.gtl.py` continue to work via the default.

## Compiler Contract

### Input

The compiler reads a single Python file: `genesis.gtl.py` at the project root.

```
genesis.gtl.py
├── Operator declarations
├── Edge declarations
├── Graph class (and optional subclasses for domain extensions)
├── Profile algebra expressions
├── Composition functions (decorated with @composition)
├── Tenant declarations
└── Workspace class
```

The file is a valid Python module. The compiler imports it, inspects the declared objects, and emits artifacts. No runtime execution of the iterate function occurs during compilation.

### Compilation phases

**Phase 1 — Parse**: Import `genesis.gtl.py`, collect all declared GTL objects (Graph, Profile, Operator, Composition, Workspace, Tenant).

**Phase 2 — Resolve**: For each Graph, resolve `extends` chains. Merge operator registries. Resolve `extra_edges` against parent edge lists.

**Phase 3 — Validate**: Run all validation rules (see above per construct). Collect errors and warnings. Compilation halts on any ERROR; warnings are reported but do not halt.

**Phase 4 — Emit**: Generate output artifacts.

### Output artifacts

| GTL construct | Compiled output | Location |
|--------------|----------------|----------|
| `Graph` (primary) | `graph_topology.yml` | `.ai-workspace/graph/` |
| Each `Edge` | `{source}_{target}.yml` | `.ai-workspace/graph/edge_params/` |
| Each `Profile` | `{name}.yml` | `.ai-workspace/graph/profiles/` |
| `Composition` functions | Entries in `compositions_registry.yml` | `.ai-workspace/graph/` |
| `Workspace` | `workspace_bindings.yml` | `.ai-workspace/graph/` |
| `Tenant` objects | Entries in `workspace_bindings.yml` | `.ai-workspace/graph/` |

The content of `graph_topology.yml` and `edge_params/*.yml` is identical to the current hand-authored YAML format — the engine, genesis_monitor, and installer can consume them without change.

### Compiler validation rules (complete)

```
E001  Operator name is not unique in the graph operator registry
E002  Operator bind form is incompatible with declared category
        (e.g., F_D operator with bind="agent::*")
E003  Edge references an operator not in the graph operator registry
E004  Edge source or target not in graph nodes
E005  Edge creates a cycle in the graph DAG
E006  sequence contains a node not in graph nodes
E007  sequence is not a valid topological ordering of edges
E008  co-evolution tuple in sequence has no corresponding Edge with co_evolve=True
E009  Profile subtraction leaves empty node set
E010  Profile addition introduces new nodes without sequence_inject
E011  Profile addition creates a cycle
E012  Profile intersection produces empty edge set
E013  extends chain creates circular inheritance
E014  extra_edges redefines parent edge without override=True
E015  Tenant name collides with a graph node name
E016  Workspace has duplicate Tenant names
E017  Composition.until() condition references an operator not in the edge evaluators
E018  Composition.on_stuck() or .on_timeout() escalation target is not an F_H operator
E019  @composition function has side effects (detected via static analysis: no I/O, no mutations)

W001  Tenant impl path does not exist (new project — create directory)
W002  Profile subtraction removes a node that is a required predecessor of a retained node
W003  Edge convergence=human_approves but no F_H operator in evaluators list
W004  Graph defines no F_H operators (no human gate — advisory for regulated domains)
W005  Composition timeout is shorter than the estimated iteration wall time
```

### Invocation

```bash
# Compile genesis.gtl.py → emit YAML artifacts
python -m genesis compile

# Validate only — no artifact emission
python -m genesis compile --validate-only

# Compile to a specific output directory
python -m genesis compile --output .ai-workspace/graph/

# Show compiled topology (human-readable summary)
```

## The Three Proving Cases (Spike Validation Targets)

Per Codex `20260311T220312`: the spike validates semantic coverage of the Python DSL against these three cases, not syntax aesthetics.

### Case 1: Standard profile subtraction

Verify that a Profile derived by subtraction from a base Graph:
- Produces a valid `graph_topology.yml` matching the current `standard.yml` profile content
- Rejects invalid subtractions (E009, W002)
- The compiled `profiles/hotfix.yml` skip-list is identical to the current hand-authored `hotfix.yml`

```python
hotfix = Profile("hotfix",
    StandardGraph - {"feature_decomposition", "module_decomposition",
                     "basis_projections", "uat_tests", "cicd"})
# Expected: valid 6-node subgraph; no broken chains; skip-list in compiled YAML
```

### Case 2: Domain extension

Verify that a Graph subclass with `extra_edges` and a modified sequence:
- Compiles to a valid `graph_topology.yml` containing all parent edges plus the new edge
- The new edge parameter file is emitted to `edge_params/`
- The sequence ordering is preserved in the output topology

```python
class FintechSDLC(Graph):
    name = "fintech_sdlc"
    extends = StandardGraph
    extra_edges = [Edge("design", "compliance_review", evaluators=[...], human_required=True)]
    sequence = [..., "design", "compliance_review", "module_decomposition", ...]
# Expected: graph_topology.yml contains compliance_review node and edge
# Expected: edge_params/design_compliance_review.yml emitted
```

### Case 3: Higher-order composition

Verify that a `@composition` function:
- Compiles to a `compositions_registry.yml` entry with the correct parameter schema
- When bound to an Edge via `execute=`, the edge parameter YAML contains the composition reference
- The control flow primitives (`.until()`, `.timeout()`, `.on_stuck()`) are serialised correctly

```python
@composition
def tdd_cycle(feature, code_asset, test_asset, coverage_threshold=80, timeout_duration="2h"):
    return (
        iterate(co_evolve(code_asset, test_asset))
        .until(all_tests_pass & coverage(threshold=coverage_threshold))
        .timeout(timeout_duration)
        .on_stuck(escalate(human_approval))
    )
# Expected: compositions_registry.yml entry: {name: tdd_cycle, params: [...], control_flow: {...}}
# Expected: edge using execute=tdd_cycle has composition_ref in its edge_params YAML
```

## Migration from Static YAML

Projects with existing hand-authored YAML files are not required to migrate immediately. The compiler is additive:

1. If `genesis.gtl.py` is absent, the engine uses the existing YAML files directly (no change)
2. If `genesis.gtl.py` is present, the compiler emits YAML to the same locations, overwriting the hand-authored files
3. The installer (`gen-setup.py`) acquires a new step: if `genesis.gtl.py` exists, run `python -m genesis compile` before workspace setup

**Migration path for existing projects:**

```bash
# Generate a genesis.gtl.py from existing YAML (reverse compilation)
python -m genesis decompile --from .ai-workspace/graph/ --output genesis.gtl.py

# Verify the round-trip is lossless
python -m genesis compile --validate-only
diff .ai-workspace/graph/graph_topology.yml <(python -m genesis compile --dry-run --show-topology)
```

The `decompile` command is a best-effort reverse compiler. It produces a GTL program that, when compiled, produces YAML identical to the existing files. Human review is required before treating the decompiled output as canonical.

## Consequences

### What changes

- `genesis.gtl.py` at project root is the new canonical authoring surface
- The existing YAML files become compiled materialisations (but remain the execution contract)
- The installer acquires a compile step
- Methodology vendors publish `genesis.gtl.py` files, not YAML bundles
- The design marketplace ships GTL programs; installing a methodology means instantiating a GTL program with project values
- Profile validity is enforced at compile time, not at runtime

### What does not change

- `iterate(Asset, Context[], Evaluators)` — the only operation
- `invoke(intent, state) → StepResult` — the engine invocation contract (ADR-S-016)
- The event log (`.ai-workspace/events/events.jsonl`) — append-only source of truth
- Feature vectors — projections of the event stream
- `project_constraints.yml` — the values layer
- `genesis_monitor` — consumes compiled `graph_topology.yml` unchanged
- All existing edge_params YAML schema — the compiler emits the same schema

### What is deferred to Phase 2

- Engine consuming GTL directly (live topology traversal without compiled YAML intermediary)
- Custom GTL grammar / syntax (Phase 1 is Python DSL; Phase 2 may add syntactic sugar)
- Downgrading `graph_topology.yml` from canonical authoring contract to materialisation detail
- IDE tooling (language server, syntax highlighting for a future custom grammar)

## Open Questions (not blocking Phase 1 ratification)

1. **decompile fidelity**: The reverse compiler is best-effort. What is the accepted tolerance for round-trip differences (e.g., evaluator ordering, YAML comment preservation)?

2. **Operator versioning**: Can an operator name be versioned (`pytest@2`)? Useful when a methodology evolves and two versions of an operator coexist.

3. **GTL as spec** (Phase 2 decision): The bootstrap graph in the bootloader §XIV is already written as a graph expression. Should the Genesis specification itself be expressed in GTL, making the spec directly executable? Deferred — requires Phase 1 to prove GTL is expressive enough.

4. **Ecosystem composition**: ADR-S-034 implies Genesis-enabled services might publish their own GTL fragments that other services can import. The module import mechanism for cross-project GTL composition is not yet defined.
